## Context

当前 `SandboxExecutor` 是 Kabot 工具执行链路的公共底座，`shell`、`spawn` 以及部分依赖外部命令的工具都通过它执行宿主系统命令。现有实现采用 Unix 专属的 `fork`/`exec`/`waitpid` 与 `/bin/sh -c` 路径，在 Windows 分支中直接返回不支持，因此工具集合虽然在注册层面一致，但在运行时对不同宿主平台暴露出不同能力。

这个变更是一个跨模块改动：它不仅影响 `src/sandbox`，还会影响依赖其结果语义的工具实现和相关测试。设计需要确保不同平台下对工作目录、超时、输出采集、阻断策略和错误反馈的定义一致，并尽量减少调用方对平台细节的感知。

## Goals / Non-Goals

**Goals:**
- 为 Unix 和 Windows 提供统一的 `SandboxExecutor` 执行语义。
- 保持现有工具调用接口基本不变，让 `shell`、`spawn`、`tts` 等调用方继续通过同一入口执行命令。
- 在两个平台上都支持工作目录切换、stdout/stderr 采集、超时终止和策略阻断。
- 将平台差异收敛在沙箱层内部，而不是扩散到每个工具实现中。
- 提供覆盖关键执行路径的自动化测试，证明双平台行为一致。

**Non-Goals:**
- 不在本次变更中引入容器级或虚拟化级隔离。
- 不尝试实现完整的 shell 语法规范统一，只保证通过平台默认命令解释器可执行并返回标准化结果。
- 不重新设计工具注册机制或修改 LLM 工具协议。
- 不解决外部命令本身在两平台是否安装的问题，只要求失败时有一致、可诊断的结果。

## Decisions

### 1. 保留单一 `SandboxExecutor::Run` 接口，对内做平台分派
继续使用现有 `SandboxExecutor::Run(command, working_dir, timeout)` 作为唯一入口，避免修改所有调用方。内部按平台走不同执行实现：Unix 保留现有进程模型并重构为共享辅助逻辑；Windows 新增基于原生命令进程创建 API 的实现。

备选方案：
- 让每个工具自行处理 Windows/Unix 分支。放弃，因为会让策略判断、超时处理和输出收集重复分散。
- 用 `std::system` 或 `_popen` 统一实现。放弃，因为难以可靠处理超时、工作目录和 stdout/stderr 分离。

### 2. 统一通过平台默认命令解释器执行字符串命令
Unix 继续使用 `/bin/sh -c`；Windows 使用 `cmd.exe /C` 作为默认解释器。这样可以保持工具层仍然接受单个命令字符串，降低对现有提示词和调用方式的影响。

备选方案：
- 改为参数数组执行。更安全，但会改变工具协议并要求上层调用全面改写。
- 使用 PowerShell。能力更强，但对已有命令字符串兼容性和环境依赖更敏感。

### 3. 将阻断策略提升为平台无关的预检查层
命令阻断逻辑不应嵌在 Unix 执行分支中，而应在真正启动进程前统一完成。策略层维护一个平台无关的危险 token 集合，并允许按平台补充额外规则。被阻断时返回 `blocked=true`、非零退出码和可诊断错误信息。

备选方案：
- 保持仅 Unix 生效。放弃，因为这会导致同一危险命令在 Windows 上绕过策略。
- 引入复杂命令解析器。放弃，因为超出当前变更范围。

### 4. 标准化结果语义，调用方只依赖 `ExecResult`
规定 `ExecResult` 在两个平台上都满足以下语义：
- `exit_code` 反映最终进程退出状态；超时终止时使用统一失败退出语义。
- `timed_out` 明确表示是否因为超时被终止。
- `blocked` 明确表示是否因策略阻断而未启动。
- `output` 和 `error` 分别收集 stdout/stderr，允许任一为空。

这样 `shell`、`spawn`、`tts` 不需要知道具体平台实现细节，只消费标准化结果。

### 5. 通过测试锚定行为而不是锚定具体系统调用
测试分为两层：
- 沙箱执行器单元/集成测试：验证正常执行、工作目录、生效超时、阻断策略、stdout/stderr 分离。
- 工具级测试：验证 `shell` 和 `spawn` 在拿到标准化结果后能够返回一致的用户可见输出。

这样可以允许不同平台使用不同底层 API，但对外行为保持一致。

## Risks / Trade-offs

- [Windows 进程与终止语义不同] → 通过在设计中显式定义超时后的结果语义，并在测试中只校验合同而不校验底层实现细节。
- [命令解释器差异导致同一命令字符串跨平台不可移植] → 约束本次变更只保证沙箱能力可用，不保证任意 Unix 命令在 Windows 上自动兼容；失败时必须提供清晰 stderr/exit 信息。
- [外部工具如 `ffmpeg`/`ffplay` 在不同平台安装状态不一致] → 不隐藏底层失败，保留原始 stderr，并确保工具层不要把“命令不存在”误报为沙箱不支持。
- [阻断 token 规则过于宽松或过于保守] → 先保留最小必要规则集，并通过平台测试与后续迭代微调。
- [异步 `spawn` 在 Windows 上日志与退出状态采集更复杂] → 保持后台线程模型不变，只要求线程内部调用统一沙箱接口并记录标准化结果。

## Migration Plan

1. 重构 `SandboxExecutor`，抽离共享的结果处理、策略检查与临时输出文件管理。
2. 为 Windows 增加命令执行分支，并保证工作目录、超时和 stdout/stderr 收集行为满足统一合同。
3. 校准 `shell`、`spawn`、`tts` 等调用方对错误、超时、阻断结果的消费方式，确保不会依赖旧的 Unix-only 假设。
4. 增加跨平台测试；在支持的平台上分别执行验证。
5. 如新实现出现回归，可回退到此前 Unix 实现并保留 Windows 为受限模式，但应避免合入未覆盖测试的半成品实现。

## Open Questions

- Windows 分支默认是否固定使用 `cmd.exe /C`，还是允许未来配置成 PowerShell。
- 是否需要把被阻断命令返回的 `exit_code` 固定成一个明确约定值，便于上层进一步处理。
- 当前测试框架对 Windows 平台命令执行的集成测试支持程度如何，是否需要引入条件编译测试或平台专用测试用例。
